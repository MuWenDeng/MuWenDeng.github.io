3. 拦截器（Interceptor）与过滤器（Filter）对比

DispatcherServlet负责：将接收到的HTTP请求，发给相应的控制器处理

常见面试问题

过滤器在DispatcherServlet之前执行，拦截器在DispatcherServlet之后执行

多个过滤器按注册顺序执行，多个拦截器按配置顺序执行

直接使用@Autowired无效，需要通过WebApplicationContextUtils获取

或使用Spring的DelegatingFilterProxy

3）什么情况下拦截器的postHandle方法不会执行？

preHandle返回false时

Controller抛出异常且未被@ControllerAdvice捕获时

4）如何全局捕获Controller异常？

使用@ControllerAdvice + @ExceptionHandler（在拦截器之后执行）

无法在过滤器中捕获Controller异常

5）如何处理静态资源？

过滤器可以处理所有静态资源

拦截器默认不处理静态资源，需通过配置排除

5. 事务管理

5.1 @Transactional的工作原理
底层实现 - AOP代理，执行流程：

1）代理创建：Spring启动时，为标注@Transactional的类生成动态代理（JDK动态代理或CGLIB）
2）拦截执行：调用方法时，代理先执行事务管理逻辑

3）事务管理：
    开启事务（获取数据库连接，设置autocommit=false）
    执行业务方法
    提交事务或回滚事务

4）异常处理：遇到RuntimeException/Error时回滚，正常执行则提交


理解代理：
老板（真实对象）：专注于核心业务（处理重要事务）
秘书（代理对象）：帮老板处理杂务（安排会议、接电话、记录日程）


静态代理（手动编写代理类）：要为每个类都写一个代理类，工作量巨大。如果要为100个Service添加事务，要写100个代理类。
动态代理（运行时自动生成代理类）：在程序运行时，动态地为目标类生成代理类。


@Transactional的实现机制是：Spring基于AOP思想，使用TransactionInterceptor这个具体的拦截器来实现事务管理。

说它是Aspect：因为它遵循AOP的横切关注点思想，将事务逻辑从业务代码中分离，通过代理模式实现。

说它是Interceptor：因为具体实现类TransactionInterceptor实现了AOP联盟标准的MethodInterceptor接口，使用标准的拦截器模式在执行目标方法前后添加事务逻辑。

更准确的说法：@Transactional是Spring AOP框架的一个具体应用，底层通过TransactionInterceptor拦截器实现，提供声明式事务管理功能。它利用了Spring的代理机制（JDK动态代理或CGLIB），在运行时为标注@Transactional的Bean创建代理对象，通过拦截器添加事务管理逻辑。



Spring如何选择JDK代理还是CGLIB代理：

// Spring Boot的配置
@Configuration
@EnableAspectJAutoProxy(proxyTargetClass = true)  // true=CGLIB, false=JDK
public class AppConfig {
}

// 或者application.properties配置
// spring.aop.proxy-target-class=true  # 强制使用CGLIB

**选择策略：**
默认情况：
目标类实现了接口 → 使用JDK动态代理
目标类没有实现接口 → 使用CGLIB动态代理

强制使用CGLIB：
设置proxyTargetClass = true
统一使用CGLIB，避免类型转换问题

5.2 事务的传播行为（Propagation）
传播行为定义了多个事务方法相互调用时，事务如何传播。

7种传播行为详解
import org.springframework.transaction.annotation.Propagation;
import org.springframework.transaction.annotation.Transactional;

@Service
public class OrderService {
    
    @Autowired
    private UserService userService;
    
    // REQUIRED（默认）：如果当前存在事务，则加入该事务；如果不存在，则创建新事务
    @Transactional(propagation = Propagation.REQUIRED)
    public void placeOrder(Order order) {
        // 方法A的事务
        saveOrder(order);
        
        try {
            // 调用另一个事务方法
            userService.updateUserPoints(order.getUserId(), order.getPoints());
        } catch (Exception e) {
            // 异常处理
        }
        // REQUIRED：updateUserPoints加入placeOrder的事务
        // 如果updateUserPoints抛出异常，整个事务回滚（包括saveOrder）
    }
}

@Service
public class UserService {
    
    // REQUIRES_NEW：无论当前是否存在事务，都创建新事务
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void updateUserPoints(Long userId, Integer points) {
        // 创建新事务，与调用方的事务独立
        // 自己的提交/回滚不影响调用方事务
    }
    
    // NESTED：如果当前存在事务，则在嵌套事务内执行
    @Transactional(propagation = Propagation.NESTED)
    public void logUserAction(UserAction action) {
        // 嵌套事务：外部事务回滚会导致嵌套事务回滚
        // 但嵌套事务可以独立回滚而不影响外部事务
    }
    
    // SUPPORTS：如果当前存在事务，则加入事务；如果不存在，则以非事务方式执行
    @Transactional(propagation = Propagation.SUPPORTS)
    public User getUser(Long id) {
        // 查询方法，不需要强制事务
    }
    
    // NOT_SUPPORTED：以非事务方式执行，如果当前存在事务则挂起
    @Transactional(propagation = Propagation.NOT_SUPPORTED)
    public void sendNotification(Notification notification) {
        // 发消息、日志等非数据库操作
    }
    
    // MANDATORY：强制必须在事务中调用，否则抛出异常
    @Transactional(propagation = Propagation.MANDATORY)
    public void auditLog(AuditLog log) {
        // 必须在外层事务中调用
    }
    
    // NEVER：必须在非事务环境下执行，否则抛出异常
    @Transactional(propagation = Propagation.NEVER)
    public String generateReport() {
        // 生成报表，不希望被事务影响性能
    }
}

5.4 声明式事务与编程式事务的区别。

特性	声明式事务	编程式事务
实现方式	通过注解或XML配置	通过编程代码手动控制
侵入性	非侵入式（不影响业务逻辑）	侵入式（与业务代码耦合）
核心组件	@Transactional 注解、AOP代理	TransactionTemplate、PlatformTransactionManager
使用场景	大多数业务场景	需要精细控制的复杂场景

代码简洁性	✅ 简洁，一行注解	❌ 代码冗长，需要模板代码
业务逻辑分离	✅ 事务管理与业务逻辑完全分离	❌ 事务代码与业务逻辑混合
灵活性	❌ 相对固定，配置有限	✅ 极高灵活性，可动态控制
可读性	✅ 清晰直观，关注业务	❌ 事务代码干扰业务逻辑
性能	稍有开销（AOP代理）	直接控制，开销较小
错误处理	自动回滚（声明异常）	可精细控制回滚时机
适用场景	标准CRUD、常规业务	复杂业务流、需要条件提交/回滚

声明式事务（工作原理：AOP代理）：
@Service
public class UserService {
    
    // 最简单的声明式事务
    @Transactional
    public void updateUser(User user) {
        // 业务逻辑
        userRepository.save(user);
        // 如果抛出异常，事务自动回滚
    }
    
    // 带配置的声明式事务
    @Transactional(
        propagation = Propagation.REQUIRED,
        isolation = Isolation.DEFAULT,
        rollbackFor = Exception.class,
        timeout = 30
    )
    public void batchUpdate(List<User> users) {
        // 批量操作
    }
}

编程式事务，例如TransactionTemplate方式：
@Service
public class OrderService {
    
    @Autowired
    private TransactionTemplate transactionTemplate;
    
    public void complexOrderProcess(Order order) {
        // 使用TransactionTemplate执行事务
        transactionTemplate.execute(status -> {
            try {
                // 步骤1：创建订单
                orderRepository.save(order);
                
                // 步骤2：扣减库存
                inventoryService.deduct(order);
                
                // 步骤3：记录日志
                logService.record(order);
                
                // 手动设置只读（如果需要）
                // status.setRollbackOnly(); // 手动回滚
                
                return true;
            } catch (Exception e) {
                // 发生异常，事务会自动回滚
                // 也可以手动回滚：status.setRollbackOnly();
                throw e;
            }
        });
    }
}

可恢复性	✅ 可恢复（通过捕获处理）	❌ 不可恢复（严重系统问题）

你的maximumPoolSize怎么确定的？
1） 理论基准（展示知识）：
   "我会参考多个因素，而不是简单套用公式。首先会基于CPU核心数作为起点，
   现代SSD环境下，磁盘数不再重要，更多考虑并发特性。"

2） 实际方法（展示经验）：
   "在实际项目中，我会：
   a) 先用保守值（如20）启动
   b) 进行压测，观察性能拐点
   c) 监控生产环境连接使用率
   d) 根据业务增长动态调整"

3） 具体数值（展示专业性）：
   "对于我们的电商订单服务：
   - CPU：16核
   - 理论最大：16×8=128
   - 实际配置：80（留出buffer）
   - 监控显示：峰值使用60，平均30"

4） 安全考虑（展示周全性）：
   "同时我会确保：
   - 不超过数据库max_connections的80%
   - 设置合理的connection-timeout防止线程阻塞
   - 开启leak-detection检测连接泄露"

Step 1：JDBC URL 加一个关键参数
yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/test_db?
         useSSL=false&
         serverTimezone=GMT%2B8&
         rewriteBatchedStatements=true    # ← 这个参数是灵魂！
    username: root
    password: 123456
rewriteBatchedStatements=true 的作用：

没有这个参数：JDBC 的 executeBatch() 会拆成多条单条 INSERT 发送，性能差

加上这个参数：MySQL JDBC 驱动会把多条 INSERT 重写成一条多值 INSERT，性能提升 10~20 倍

Step 2：使用 SqlSession 执行批量提交
java
@Service
@Slf4j
public class UserBatchService {

    @Autowired
    private SqlSessionFactory sqlSessionFactory;

    public void batchInsert(List<User> users) {
        // 每批 2000 条，避免单条 SQL 太长
        int batchSize = 2000;
        int total = users.size();

        try (SqlSession sqlSession = sqlSessionFactory.openSession(ExecutorType.BATCH, false)) {
            UserMapper mapper = sqlSession.getMapper(UserMapper.class);

            for (int i = 0; i < total; i++) {
                mapper.insert(users.get(i));  // 这里不会立即执行，而是攒在批里
                
                if (i % batchSize == batchSize - 1 || i == total - 1) {
                    sqlSession.flushStatements();  // 每 2000 条发送一次
                    sqlSession.commit();            // 提交事务
                    log.info("已插入 {} 条", i + 1);
                }
            }
        }
    }
}
Step 3：Mapper 中只需要普通的 insert
java
@Mapper
public interface UserMapper extends BaseMapper<User> {
    // 普通的单条插入
    @Insert("INSERT INTO users (name, age, email) VALUES (#{name}, #{age}, #{email})")
    int insert(User user);
}

8. RabbitMQ
8.1 RabbitMQ 防消息丢失
第一阶段：生产端（Producer）—— 确保消息“发出去了”
这一阶段最容易丢消息，比如网络抖动、或 RabbitMQ 服务挂了导致写入失败。

解决办法：开启 Confirm 确认机制 + 重试

核心思想：Broker（RabbitMQ）收到消息后，必须给我回一个“ACK”确认包，否则我就重发。

java
// 1. 在 application.yml 中开启 Confirm
spring:
  rabbitmq:
    publisher-confirm-type: correlated  # 开启 Confirm 机制
    publisher-returns: true             # 开启 Return 机制（路由失败回调）

// 2. 编写发送代码（带回调监听）
@Service
public class MessageProducer {

    @Autowired
    private RabbitTemplate rabbitTemplate;

    public void send(String exchange, String routingKey, Object message) {
        // 设置 CorrelationData（包含唯一ID，用于追踪）
        CorrelationData correlationData = new CorrelationData(UUID.randomUUID().toString());

        // 发送消息，并设置 ConfirmCallback
        rabbitTemplate.confirmCallback = (correlation, ack, cause) -> {
            if (!ack) {
                // 失败！记录日志，写入本地数据库或文件，定时任务补发
                System.err.println("消息发送失败，correlationId: " + correlation + "，原因：" + cause);
                // 在这里触发你的重试机制（存库 + 定时任务扫描重发）
            } else {
                System.out.println("消息成功到达 Broker");
            }
        };

        // 发送消息
        rabbitTemplate.convertAndSend(exchange, routingKey, message, correlationData);
    }
}
终极保障： 生产环境必须配合 本地消息表。如果 Confirm 返回失败，直接把这笔消息 Insert 到本地 MySQL 表中，由定时任务（每分钟扫一次）重新发送，直到状态变为“已发送”。

第二阶段：MQ 服务端（Broker）—— 确保消息“不丢在内存里”
RabbitMQ 默认收到消息是存内存的，如果此时服务器突然宕机，消息就丢了。

解决办法：开启持久化（Durable） + 镜像队列（Mirror）

交换机、队列、消息都必须持久化：

声明队列时设置 durable=true。

发送消息时设置 MessageProperties.PERSISTENT_TEXT_PLAIN（Spring Boot 默认是持久化，但建议显式设置）。

（关键）开启镜像队列：

单节点 RabbitMQ 即使开启了持久化，数据写入硬盘有延迟（默认每秒刷盘几次），断电瞬间可能来不及刷盘。

生产方案：搭建集群，并配置 镜像队列（Mirror Queue）。即消息写入 Master 节点时，会自动同步到 1-2 个 Slave 节点。即使 Master 挂了，Slave 秒变 Master，消息完整无缺。

第三阶段：消费端（Consumer）—— 确保消息“处理完再确认”
这是最容易忽略的坑：消费者拿到消息，还没处理完业务逻辑，就自动回复 ACK 了。此时消费者突然宕机，消息已从队列删除，永久丢失。

解决办法：关闭自动 ACK，改为手动签收（Manual Ack）

java
// 1. 配置 application.yml
spring:
  rabbitmq:
    listener:
      simple:
        acknowledge-mode: manual  # 关键！关闭自动，改手动

// 2. 消费代码
@Component
public class MessageConsumer {

    @RabbitListener(queues = "order_queue")
    public void handleMessage(Message message, Channel channel, @Header(AmqpHeaders.DELIVERY_TAG) long tag) {
        try {
            // 1. 处理业务逻辑（操作数据库、调用第三方接口等）
            System.out.println("处理消息: " + new String(message.getBody()));
            
            // 2. 业务处理成功！手动发送 ACK
            channel.basicAck(tag, false); 
            System.out.println("消息处理成功，已确认");

        } catch (Exception e) {
            // 3. 处理失败！根据情况决定是否重试
            try {
                // 场景A：业务异常，重试，重新放回队列（根据业务决定是否延迟重试）
                // requeue=true 重新入队，会一直重试，可能造成死循环
                // 建议：重试3次后，打入死信队列（DLX），人工介入处理
                channel.basicNack(tag, false, false); // false表示不重新入队，打入死信
                System.err.println("消息异常，已转入死信队列");
            } catch (Exception ex) {
                ex.printStackTrace();
            }
        }
    }
}
生产避坑：basicNack 的第三个参数 requeue 要谨慎设置。生产环境通常设为 false，配合 死信队列（DLX） 专门存放处理失败的消息，人工补偿处理。如果设为 true，异常消息会无限循环，堵塞整个队列。


8. 定时任务

方式一：@Scheduled 注解（最常用）
2.1 开启定时任务
在启动类或配置类上加 @EnableScheduling：

java
@SpringBootApplication
@EnableScheduling  // ← 开启定时任务
public class DemoApplication {
    public static void main(String[] args) {
        SpringApplication.run(DemoApplication.class, args);
    }
}
2.2 基本用法
java
@Component
@Slf4j
public class ScheduledTasks {

    // 1. 固定延迟：上次执行完后延迟 5 秒再执行
    @Scheduled(fixedDelay = 5000)
    public void taskWithFixedDelay() {
        log.info("固定延迟任务执行: {}", LocalDateTime.now());
    }

    // 2. 固定频率：每 10 秒执行一次（不关心上次是否完成）
    @Scheduled(fixedRate = 10000)
    public void taskWithFixedRate() {
        log.info("固定频率任务执行: {}", LocalDateTime.now());
    }

    // 3. Cron 表达式：每天凌晨 2 点执行
    @Scheduled(cron = "0 0 2 * * ?")
    public void dailyReportTask() {
        log.info("每日报表生成开始...");
        generateDailyReport();
    }

    // 4. Cron 表达式：每小时执行一次
    @Scheduled(cron = "0 0 * * * ?")
    public void hourlyTask() {
        log.info("每小时任务执行");
    }

    // 5. 动态参数：从配置文件读取 cron 表达式
    @Scheduled(cron = "${schedule.report.cron:0 0 2 * * ?}")
    public void configurableTask() {
        log.info("可配置的定时任务执行");
    }
}

方式二：异步定时任务（@Async + @Scheduled）
默认情况下，所有 @Scheduled 方法是同一个线程串行执行的。如果一个任务执行时间过长，会阻塞后续任务。

配置异步调度：

java
@Configuration
@EnableAsync  // 开启异步
public class AsyncScheduledConfig {

    @Bean(name = "taskExecutor")
    public Executor taskExecutor() {
        return new ThreadPoolExecutor(
            5, 20, 60L, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(100),
            new ThreadFactoryBuilder()
                .setNameFormat("scheduled-pool-%d")
                .build()
        );
    }
}
使用异步定时任务：

java
@Component
@Slf4j
public class AsyncScheduledTasks {

    @Async("taskExecutor")  // ← 使用指定线程池异步执行
    @Scheduled(fixedRate = 5000)
    public void asyncTask() {
        log.info("异步定时任务执行，线程: {}", Thread.currentThread().getName());
        // 即使这个任务执行 10 秒，也不会阻塞其他任务
    }
}

方式三：分布式定时任务（ShedLock）
问题：微服务多实例部署时，同一个定时任务会在多个实例上重复执行。

解决方案：使用 ShedLock，通过 Redis 或数据库实现分布式锁。

5.1 添加依赖
xml
<dependency>
    <groupId>net.javacrumbs.shedlock</groupId>
    <artifactId>shedlock-spring</artifactId>
    <version>5.10.0</version>
</dependency>
<dependency>
    <groupId>net.javacrumbs.shedlock</groupId>
    <artifactId>shedlock-provider-redis-spring</artifactId>
    <version>5.10.0</version>
</dependency>
5.2 配置 ShedLock
java
@Configuration
@EnableScheduling
@EnableSchedulerLock(defaultLockAtMostFor = "PT30S")  // 默认锁持有时间
public class ShedLockConfig {

    @Bean
    public LockProvider lockProvider(RedisConnectionFactory connectionFactory) {
        return new RedisLockProvider(connectionFactory);
    }
}
5.3 使用 @SchedulerLock 注解
java
@Component
@Slf4j
public class DistributedScheduledTasks {

    @Scheduled(cron = "0 0 2 * * ?")
    @SchedulerLock(
        name = "dailyReportTask",
        lockAtLeastFor = "PT5M",   // 锁至少持有 5 分钟
        lockAtMostFor = "PT10M"    // 锁最多持有 10 分钟
    )
    public void dailyReportTask() {
        log.info("分布式锁保护的任务执行，只有持有锁的实例执行");
        generateDailyReport();
    }
}
原理：只有一个实例能获得 Redis 锁并执行任务，其他实例跳过。

方式四：分布式调度平台 (XXL-JOB)（微服务与复杂任务的首选）
在微服务或中大型项目中，XXL-JOB 是目前国内生产环境最主流的分布式任务调度平台之一。它解决了 @Scheduled 在多实例下的所有痛点。

核心优势：

统一管理：提供Web可视化界面，可集中管理所有任务，动态配置Cron表达式，无需重启应用。

高可用：支持调度中心集群部署，避免单点故障。

强大的执行能力：拥有任务分片、失败重试、路由策略（如轮询、故障转移）、任务依赖等生产级特性。


9. 最常见的 Spring Web API 内存泄漏场景
一、ThreadLocal 未清理（Spring 框架级经典）

真实代码（出问题版本）
```java
@RestController
public class UserController {
    // 自定义 ThreadLocal 保存请求追踪 ID
    private static final ThreadLocal<String> traceIdHolder = new ThreadLocal<>();
    
    @GetMapping("/api/user")
    public User getUser() {
        String traceId = UUID.randomUUID().toString();
        traceIdHolder.set(traceId);          // 设置
        // ... 业务逻辑 ...
        return userService.getUser();
        // ❌ 没有调用 traceIdHolder.remove()
    }
}
```

后果
随着请求量增大，Tomcat 线程池中每个线程都会累积一个 `traceId` 字符串引用，这些字符串永远不会被 GC 回收。内存持续增长，最终 OOM。

解决方案
```java
try {
    traceIdHolder.set(traceId);
    // 业务逻辑
} finally {
    traceIdHolder.remove();   // ✅ 确保清理
}
```

二、Stream / 大文件 / 图片处理未释放资源

场景
API 需要导出 Excel、处理图片、或读取大文件流。**流没有正确关闭**，导致堆外内存（Direct Memory）或文件句柄泄漏。堆外内存泄漏不会显示在 Heap Dump 中，非常隐蔽。

真实代码（出问题版本）
```java
@GetMapping("/api/export")
public ResponseEntity<byte[]> exportData() {
    FileInputStream fis = new FileInputStream("/tmp/large.xlsx");
    byte[] data = new byte[fis.available()];  // ❌ 直接读全部到内存
    fis.read(data);
    // ❌ fis 没有 close
    return ResponseEntity.ok(data);
}
```

后果
- 如果文件很大（100MB），每次请求都会在堆里创建 100MB 的 `byte[]`，同时 `FileInputStream` 持有的文件句柄不释放，最终 OOM 或 `Too many open files`。
- 如果使用了 Netty / Tomcat NIO 的 DirectBuffer，还会耗尽堆外内存（`java.lang.OutOfMemoryError: Direct buffer memory`）。

解决方案
```java
// 方案1：使用 try-with-resources
try (FileInputStream fis = new FileInputStream("/tmp/large.xlsx");
     ByteArrayOutputStream baos = new ByteArrayOutputStream()) {
    byte[] buffer = new byte[8192];
    int len;
    while ((len = fis.read(buffer)) != -1) {
        baos.write(buffer, 0, len);
    }
    return ResponseEntity.ok(baos.toByteArray());
}

// 方案2：流式响应（不把整个文件加载到内存）
@GetMapping("/api/export-stream")
public ResponseEntity<StreamingResponseBody> exportStream() {
    StreamingResponseBody body = outputStream -> {
        try (FileInputStream fis = new FileInputStream("/tmp/large.xlsx")) {
            byte[] buffer = new byte[8192];
            int len;
            while ((len = fis.read(buffer)) != -1) {
                outputStream.write(buffer, 0, len);
            }
        }
    };
    return ResponseEntity.ok(body);
}
```

最后一句忠告

**预防比排障更重要**。在生产环境，一定要开启：

```bash
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/path/to/dump/
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/path/to/gc.log
```

这样一旦 OOM，自动生成 Heap Dump 和 GC 日志，你手上有 `jstack`、`jmap`、`jstat` 这些工具，就能快速定位问题源头。

如果你碰到过具体的 OOM 场景（比如某个接口调用后内存暴涨），可以把代码片段或 GC 日志发来，我帮你做一次现场分析。
