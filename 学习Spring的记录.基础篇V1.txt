1.3 volatile 修饰符
public class MyRunnable implements Runnable
{
    private volatile boolean active;
    public void run()
    {
        active = true;
        while (active) // 第一行
        {
            // 代码
        }
    }
    public void stop()
    {
        active = false; // 第二行
    }
}

多线程：
run() 在一个线程里跑
stop() 在另一个线程里调用

多线程同享同一个变量 active。这是 Java 内存模型（JMM） 的能力。

如果没有volatile，JVM会把 active 缓存在CPU寄存器里，导致while永远读不到false。volatile保证不缓存，每次从主内存读。

分析：
寄存器缓存是编译器/JIT的优化行为，目的是减少昂贵的内存访问。
volatile通过内存屏障阻止这种优化，保证每次从主内存读取。
问题不只是寄存器，还包括CPU缓存一致性、写缓冲区等。


从字节码理解 count++ 的三次独立操作
在 Java 中，count++ 看起来像一行代码，但编译成字节码后，它是三个独立的操作：
1）读（Read）：从主内存中获取 count 的当前值到工作内存。
2）改（Modify）：在工作内存中将值加 1。
3）写（Write）：将新的值从工作内存写回主内存。
这三次操作之间，为线程切换留下了间隙。

多线程环境下 volatile 变量 count++ 操作丢失更新问题
1）主内存中 count 初始值为 0。
2）线程 A 从主内存读取 count 的值，读到的值为 0，并将其存入线程 A 的工作内存。
3）线程 B 也从主内存读取 count 的值，读到的值为 0，并将其存入线程 B 的工作内存。
4）线程 A 在自己的工作内存中执行计算：0 + 1 = 1。
5）线程 B 也在自己的工作内存中执行相同的计算：0 + 1 = 1。
6）线程 A 将计算结果 1 写回主内存，主内存中的 count 被更新为 1（volatile 保证了立即可见性）。
7）随后，线程 B 也将自己的计算结果 1 写回主内存，主内存中的 count 被覆盖为 1。
结果说明： 虽然执行了两次自增操作，但最终结果仅为 1，丢失了一次更新。这说明了 volatile 虽然能保证可见性，但无法保证复合操作（如 count++）的原子性。


要解决count++这个问题，就必须把这三步变成一个原子操作。在 Java 中，有两种标准做法：

1. 使用 synchronized（悲观锁）
public class Counter {
    private volatile int count = 0;
    public synchronized void increment() {
        count++; // 现在整个方法被锁保护，其他线程无法进入
    }
}

2. 使用 AtomicInteger（乐观锁 / CAS）
public class Counter {
    private AtomicInteger count = new AtomicInteger(0);
    public void increment() {
        count.incrementAndGet(); // 基于CAS的无锁原子操作
    }
}

AtomicInteger 利用硬件支持的 CAS（比较并交换） 指令，在硬件层面保证了“读-改-写”是一个原子操作。如果在修改时发现值被改过，它会自动重试，从而保证最终一致性。



计算机存储层次结构：
CPU性能：寄存器 > L1缓存 > L2缓存 > L3缓存 > 主内存 > 磁盘
访问速度：0.5ns   > 1ns   > 3ns   > 10ns   > 100ns  > 10ms
容量大小：KB级别  < MB级别 < MB级别 < MB级别 < GB级别 < TB级别

JVM不会把整个变量"存储"在寄存器里长期保存。准确的说法是：
// 误解：把active变量存储在寄存器里
寄存器[active] = true;  // ❌ 不是这样！

// 实际：把active的值加载到寄存器进行运算
boolean temp = active;  // 加载到寄存器
while (temp) {          // 使用寄存器中的值
    // 循环体
}
// 之后寄存器可能被重用


"寄存器缓存"的真正含义，不是物理存储，而是访问路径优化：
没有优化的情况（volatile保证）：
循环开始 → 读取内存 → CPU缓存 → 寄存器 → 判断 → 循环体
      ↑                                   │
      └───────────────────────────────────┘

优化后的情况：
      ┌───────────────────────────────────┐
      ↓                                   │
循环开始 → 寄存器(已有值) → 判断 → 循环体
      ↑                                   │
      └───────────────────────────────────┘
      
关键：优化后完全绕过了"读取内存"这个步骤！

1.4 synchronized 修饰符
synchronized 关键字声明的方法同一时间只能被一个线程访问。

保证了原子性，通过 “互斥锁” 机制，确保同一时刻只有一个线程能进入同步代码块或方法。原子性的意思是不可分割了，读→改→写 三步是一个整体，都要执行完了才释放锁。

实例
class Account {
    private int balance = 100;

    public synchronized void draw(int amount) {
        if (balance > amount) {
            System.out.println("正在取钱");
            try {
                Thread.sleep(1000);
            } catch (InterruptedException e) {}
            balance -= amount;
            System.out.println("取完钱，余额为：" + balance);
        } else {
            System.out.println("余额不足，余额为：" + balance);
        }
    }
}

// 测试多线程
class TestSynchronized {
    public static void main(String[] args) {
        Account account = new Account();

        Runnable r1 = new Runnable() {
            @Override
            public void run() {
                account.draw(80);
            }
        };

        Thread t1 = new Thread(r1);
        Thread t2 = new Thread(r1);

        t1.start();
        t2.start();
    }
}

线程安全的输出：
正在取钱
取完钱，余额为：20
余额不足，余额为：20

如果去掉 synchronized，则可能输出：
正在取钱
正在取钱
取完钱，余额为：-60
取完钱，余额为：-60

正在取钱
正在取钱
取完钱，余额为：20
取完钱，余额为：20

代码分析：synchronized 已经保证了balance互斥 + 可见性。所以不需要加上volatile。
当线程退出 synchronized 方法/块时，会把对共享变量的修改 刷新到主内存。
当其他线程进入 synchronized 方法/块时，会 重新从主内存读取最新值。

1.7
| 内存区域                              | 存储内容                                   |
| --------------------------------- | -------------------------------------- |
| **堆（Heap）**                       | `new` 创建的对象、对象里的成员变量、数组、**字符串常量池（String Pool）** |
| **栈（Stack）**                      | 局部变量、方法参数、**基本类型值**、对象引用，例如boolean b = true; b 和 true 都在 栈               |
| **运行时常量池（Runtime Constant Pool）** | 编译期常量、符号引用（类/方法/字段的元数据引用）              |
| **元空间（Metaspace）**                | 类元数据、方法信息、**静态变量（static 字段）**          |


大例子：
public class MemoryExample {

    // ------------------- 元空间 -------------------
    // 静态字段，存储在元空间
    public static int staticCounter = 0;

    // 编译期常量，存储在元空间 + 运行时常量池
    public static final double PI = 3.14159;
    public static final String GREETING = "Hello, JVM!";

    // ------------------- 对象字段（堆） -------------------
    private int instanceValue;            // 普通对象字段 → 堆
    private volatile boolean flag;        // volatile 对象字段 → 堆
    private String instanceString;        // 对象字段引用 → 堆（引用在堆，指向 String Pool 或堆）

    public MemoryExample(String str) {
        this.instanceValue = 42;
        this.flag = true;
        this.instanceString = str; // 可以指向常量池里的字符串
    }

    public void runExample() {
        // ------------------- 栈 -------------------
        int localVar = 10;                // 栈
        boolean localFlag = false;        // 栈
        String localString = "Local String"; // 栈里引用，指向常量池或堆

        MemoryExample localObj = new MemoryExample("Instance String"); 
        // localObj 引用在栈，MyExample 对象在堆
    }

    public static void main(String[] args) {
        // 栈
        MemoryExample example = new MemoryExample(GREETING); // example 引用在栈，对象在堆
        example.runExample();
    }
}

| 内存区域                              | 会不会被 GC？ | 原因 / 说明                                                                       |
| --------------------------------- | -------- | ----------------------------------------------------------------------------- |
| **堆（Heap）**                       | ✅ 会      | 堆是垃圾回收的主要战场。**所有 `new` 出来的对象**，数组，以及字符串常量池对象（String Pool）都在堆里，**无引用时 GC 回收**。 |
| **栈（Stack）**                      | ❌ 不会     | 栈是方法调用的临时空间，局部变量、方法参数、基本类型值、对象引用都在栈中，**栈帧出栈时自动释放**，无需 GC。                     |
| **运行时常量池（Runtime Constant Pool）** | ✅ 部分可回收  | 运行时常量池在 **类元数据里**，属于堆的一部分。常量池中的**符号引用或字面量常量**如果**没有被任何引用使用**，可以被 GC 回收。       |
| **元空间（Metaspace）**                | ✅ 部分可回收  | 元空间存储类元数据和静态变量。**类被卸载时**，对应元空间内存会被回收；静态变量只要类还在，就不会被回收。                        |


1）“在Java应用中，堆内存通常比栈内存大1-2个数量级”
因为堆存储所有对象，栈只存储基本类型和引用

2）“栈内存是线程私有的，总栈内存 = 线程数 × 栈大小”
线程过多时，栈内存可能很大

3）“堆内存是全局共享的，存储所有对象实例”
业务数据、缓存都在堆中

4）“调优时优先关注堆内存，栈内存通常不是瓶颈”
除非有深度递归或大量线程

5）“栈溢出容易发现（StackOverflowError），堆溢出更常见（OOM）”
生产环境更多处理堆内存问题

记住这个结论：在绝大多数Java应用中，堆内存远大于栈内存。栈内存通常只占总内存的很小一部分（<10%），除非应用创建了大量线程或使用了深度递归。


1.8 String / StringBuilder / StringBuffer
| 类型            | 适用场景       | 注意         |
| ------------- | ---------- | ---------- |
| String        | 少量拼接、不频繁修改 | 易读、简单      |
| StringBuilder | 循环拼接、动态修改  | 非线程安全，性能高  |
| StringBuffer  | 多线程拼接      | 同步开销大，性能稍低 |

口诀：
少量拼接用 String，大量动态拼接用 StringBuilder；多线程共享用 StringBuffer

为什么 StringBuilder 线程不安全？
StringBuilder 的方法（如 append、insert）没有加 synchronized
如果多个线程同时修改同一个 StringBuilder 对象，会出现 数据竞态（race condition）
例如拼接错乱、字符丢失、数组越界

StringBuffer 是线程安全的：它的关键方法都加了 synchronized

1.10

使用线程池和原子操作，例如多人抢票系统。

import java.util.concurrent.*;
import java.util.concurrent.atomic.AtomicInteger;

/**
 * 高级版 - 使用线程池和原子操作
 * 更高效、更专业的多线程实现
 */
public class ConcertTicketSystem {
    // 剩余票数（使用原子整数，保证原子性）
    private AtomicInteger remainingTickets = new AtomicInteger(100);
    
    // 成功抢到票的用户列表（线程安全的集合）
    private ConcurrentLinkedQueue<String> successBuyers = new ConcurrentLinkedQueue<>();
    
    // 卖票方法 - 使用原子操作
    public boolean tryBuyTicket(String buyerName) {
        // 使用原子操作的CAS（Compare and Swap）机制
        while (true) {
            int current = remainingTickets.get();
            if (current <= 0) {
                return false; // 票已售罄
            }
            
            // CAS操作：如果当前值仍是current，则减1
            if (remainingTickets.compareAndSet(current, current - 1)) {
                successBuyers.offer(buyerName); // 记录成功用户
                
                // 模拟业务处理时间
                try {
                    Thread.sleep(ThreadLocalRandom.current().nextInt(5, 15));
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
                
                System.out.printf("[%s] %s 成功抢到票，剩余票数: %d%n",
                        Thread.currentThread().getName(),
                        buyerName,
                        current - 1);
                return true;
            }
            // 如果CAS失败，说明被其他线程修改了，重试
        }
    }
    
    public static void main(String[] args) throws InterruptedException {
        ConcertTicketSystem ticketSystem = new ConcertTicketSystem();
        
        System.out.println("🎵 热门演唱会门票开售！ 🎵");
        System.out.println("总票数: 100 张");
        System.out.println("抢票人数: 1000 人");
        System.out.println("=".repeat(50));
        
        // 创建线程池（最佳实践：不要直接new Thread）
        int coreCount = Runtime.getRuntime().availableProcessors();
        ExecutorService executor = Executors.newFixedThreadPool(coreCount * 2);
        
        // 使用CountDownLatch控制并发
        CountDownLatch latch = new CountDownLatch(1000);
        
        // 记录开始时间
        long startTime = System.currentTimeMillis();
        
        // 提交1000个抢票任务
        for (int i = 1; i <= 1000; i++) {
            final int userId = i;
            executor.submit(() -> {
                try {
                    String buyerName = "用户" + userId;
                    ticketSystem.tryBuyTicket(buyerName);
                } finally {
                    latch.countDown(); // 任务完成
                }
            });
        }
        
        // 等待所有任务完成
        latch.await();
        
        // 关闭线程池
        executor.shutdown();
        executor.awaitTermination(1, TimeUnit.MINUTES);
        
        long endTime = System.currentTimeMillis();
        
        // 打印统计结果
        System.out.println("\n" + "=".repeat(50));
        System.out.println("🎉 抢票统计结果 🎉");
        System.out.println("总耗时: " + (endTime - startTime) + "ms");
        System.out.println("成功抢到票的用户数: " + ticketSystem.successBuyers.size());
        System.out.println("剩余票数: " + ticketSystem.remainingTickets.get());
        
        // 显示前10个抢到票的用户
        System.out.println("\n🏆 前10位抢到票的用户:");
        int count = 0;
        for (String buyer : ticketSystem.successBuyers) {
            if (count++ < 10) {
                System.out.println("  " + buyer);
            } else {
                break;
            }
        }
    }
}

启动时间 vs 运行时间
使用 new：
✅ 启动快：直接创建
❌ 运行慢：每次都要创建和连接

使用 Spring DI：
❌ 启动慢：需要扫描、分析、创建所有Bean
✅ 运行快：Bean是单例，复用已创建的对象

对于长期运行的服务（如Web服务器）：
启动时间影响不大，运行效率更重要！

Controller的切面例子，用来记录request和response、耗时：

package com.muwen.demoapi.web.aspect;

import com.fasterxml.jackson.core.JsonProcessingException;
import com.fasterxml.jackson.databind.ObjectMapper;
import lombok.extern.slf4j.Slf4j;
import org.aspectj.lang.JoinPoint;
import org.aspectj.lang.ProceedingJoinPoint;
import org.aspectj.lang.annotation.*;
import org.springframework.stereotype.Component;

@Slf4j // 相当于 private static final Logger log = LoggerFactory.getLogger(WebLogAspect.class);
@Aspect // 告诉 Spring 这是一个切面类，包含切入点（Pointcut）和通知（Advice）。没有 @Aspect，则@Pointcut、@Before、@Around 等注解无效
@Component // 让 Spring 扫描到该类并注册为 Bean，让 Spring 容器创建和管理该类的实例
public class WebLogAspect {
    private final ObjectMapper objectMapper;

    public WebLogAspect(ObjectMapper objectMapper) {
        this.objectMapper = objectMapper;
    }

    @Pointcut("execution(public * com.muwen.demoapi.web.controller..*(..))")
    public void webLogPointcut() {}

    /**
     * 前置通知：在目标方法执行前执行
     *
     * @param joinPoint 切点
     */
    @Before("webLogPointcut()")
    public void doBefore(JoinPoint joinPoint) {
        String methodName = joinPoint.getSignature().toShortString();
        Object[] args = joinPoint.getArgs();
        log.info("请求方法: {}, 参数: {}", methodName, args);
    }

    /**
     * 环绕通知：环绕目标方法执行
     *
     * @param proceedingJoinPoint 切点
     * @return 目标方法的返回值
     * @throws Throwable 如果目标方法抛出异常
     */
    @Around("webLogPointcut()")
    public Object doAround(ProceedingJoinPoint proceedingJoinPoint) throws Throwable {
        long startTime = System.currentTimeMillis();

        Object result = proceedingJoinPoint.proceed();

        // 输出执行时间
        log.info("Time Consuming: {} ms", System.currentTimeMillis() - startTime);

        return result;
    }

    /**
     * 后置通知：在目标方法执行后执行
     *
     * @param result 目标方法的返回值
     */
    @AfterReturning(pointcut = "webLogPointcut()", returning = "result")
    public void doAfterReturning(Object result) {
        try {
            String responseBody = objectMapper.writeValueAsString(result);
            log.info("响应结果: {}", responseBody);
        } catch (JsonProcessingException e) {
            log.warn("响应结果序列化失败", e);
            log.info("响应结果: {}", result);
        }
    }


}

AOP 原理
一句话概括：Spring AOP 是基于动态代理实现的，用于将横切关注点（如日志、事务）模块化，并以非侵入方式织入到目标方法的框架。

1）Spring AOP 和 AspectJ 有什么区别？
Spring AOP 是运行时动态代理，仅支持方法级别的织入。
AspectJ 是编译时/类加载织入，功能更强大（可织入字段、构造器等），但更复杂。

Spring AOP是AspectJ的一个功能子集。

2）通知的顺序：
@Around 开始（proceed() 之前的部分）
@Before
proceed()目标方法执行
@Around 继续（proceed() 之后的部分）
@AfterReturning（成功）或 @AfterThrowing（异常）
@After（最终通知，类似 finally）

3）Spring AOP 默认使用哪种动态代理？在什么情况下会使用 CGLIB？
默认情况下，如果目标类实现了至少一个接口，Spring AOP 使用 JDK 动态代理（基于 InvocationHandler）；如果目标类没有实现接口，则使用 CGLIB 生成子类代理。在 Spring Boot 2.x 后，可以通过 spring.aop.proxy-target-class=true 强制使用 CGLIB。

JDK动态代理基于接口，利用反射机制；CGLIB基于继承，通过字节码生成子类。

4）如何实现一个自定义的注解，并利用 AOP 使其生效？
定义注解 @MyLog。
创建切面类，用 @Aspect 和 @Component 标注。
在通知注解（如 @Around）中，使用 @annotation(myLog) 绑定自定义注解。
在通知方法中编写增强逻辑。

5）在微服务中，AOP 可以应用于哪些场景？
全局异常处理：@ControllerAdvice 本身就是 AOP 思想的体现。
分布式链路追踪：在切面中向请求头注入或提取 TraceId。
接口限流与熔断：通过自定义注解和 AOP 统一实现。
Feign 客户端增强：统一添加认证头、记录日志。

6）AOP 对性能有影响吗？如何优化？
影响：动态代理会带来微小的性能开销（方法调用转发）。

优化：精确切入点表达式（避免过于宽泛的 execution(* com..*.*(..))），在非必要的地方避免使用 AOP，对于性能极度敏感的场景可以考虑 AspectJ 编译时织入。

7）你在项目中是如何使用 AOP 解决实际问题的？
我在XX项目中，利用自定义注解 @OperationLog 和 AOP 实现了操作日志的统一记录。切面会拦截所有带该注解的方法，自动将操作人、时间、参数和结果异步存入数据库。这使业务代码非常干净，并且便于后续的审计和排查问题。

         ↓

HashMap 原理
问题1：HashMap的底层数据结构是什么？
Java 8+：数组 + 链表 + 红黑树

关键参数：
默认初始容量：16
负载因子：0.75
树化阈值：链表长度≥8 且 数组长度≥64时，链表转为红黑树
退化阈值：树节点≤6，红黑树退化为链表

是的，您说得对。在正常的业务数据和良好的哈希函数下，HashMap几乎不会触发红黑树转换，根据泊松分布，链表长度达到8的概率只有约一亿分之六。

但JDK引入红黑树机制主要是一种防御性编程策略，核心目的是：

防止哈希碰撞攻击：确保即使遭遇恶意攻击，性能也不会从O(1)恶化为O(n)，而是维持在O(log n)。

为劣质hashCode兜底：对开发者不规范的hashCode()实现提供一定的性能保障。

所以，虽然它是一个‘备胎’，但却是保障HashMap在极端情况下仍能保持可用性的关键安全网。在我们日常开发中，确保作为Key的对象正确实现hashCode()和equals()方法，就是避免触发这个‘备胎机制’的最佳实践。

Q1：HashMap为什么线程不安全？
回答要点：
1）多线程扩容时可能形成循环链表（JDK 1.7及之前），导致CPU 100%。
2）多线程put可能覆盖数据。（两个线程同时插入新节点，会有B覆盖A的情况）
3）size计数不准确（非原子操作）。（两个线程拿到的当前size = 5，都认为size = 5+1 = 6，实际上应该是 7 了。）

Q2：HashMap的负载因子为什么是0.75？
回答要点：
1）空间与时间的权衡：0.75在统计学上是最佳平衡点。
2）泊松分布：链表长度≥8的概率极低（约0.00000006）。
3）Spring Boot应用：可以根据实际场景调整，如缓存密集的应用可降低负载因子。

Q3：Spring Boot中如何正确使用HashMap？
回答要点：

1）优先使用ConcurrentHashMap处理共享数据。
2）考虑使用Guava Cache或Caffeine作为缓存实现。
3）Rest API返回时使用DTO而非直接返回HashMap。
4）配置类中的Map可使用@ConfigurationProperties自动绑定。

Q4：负载因子为什么默认是0.75？
空间与时间的权衡，背后的原理是泊松分布：
负载因子高（如1.0）：空间利用率高，但哈希冲突增加，查询变慢。像挤满人的电梯，空间用足了，但谁都动不了，效率极低。
负载因子低（如0.5）：冲突少，查询快，但空间浪费，扩容频繁。像空旷的停车场，停车容易，但土地闲置太多，不经济。

0.75 像高峰期的地铁，有一定拥挤，但整体运行效率最高。

Q5：如果key为null，存放在哪里？
如果key = null，则为0。根据源码：hash = (key == null) ? 0 : hash(key)

Q6：HashMap和ConcurrentHashMap有什么区别？

对比维度	HashMap	ConcurrentHashMap (JDK 1.8)
线程安全	不安全	安全
锁机制	无锁	分段锁 + CAS + synchronized（锁单个桶）
性能	高（无锁）	较高（锁粒度细）
迭代器	快速失败 (fail-fast)	弱一致性 (weakly consistent)
Null值	允许	不允许键或值为null（避免歧义）
使用场景	单线程、局部变量、只读共享	多线程高并发环境

Q1：B+Tree 对比 B-Tree
特性	B-Tree	B+Tree (MySQL采用)
数据存储	所有节点都存数据	只有叶子节点存数据，非叶子节点纯索引

事务隔离级别

隔离级别	脏读	不可重复读	幻读	实现机制	性能
读未提交
(Read Uncommitted)	✅ 可能	✅ 可能	✅ 可能	几乎无锁	最高

读已提交
(Read Committed)	❌ 避免	✅ 可能	✅ 可能	语句级快照	高

可重复读
(Repeatable Read)	❌ 避免	❌ 避免	✅ 可能	事务级快照 + Gap Lock (MySQL)	中

串行化
(Serializable)	        ❌ 避免	❌ 避免	❌ 避免	完全加锁	最低


脏读：读到其他事务未提交的数据。
不可重复读：同一个事务内，两次读取同一行数据，结果不同（因为其他事务修改并提交了）。
幻读：同一个事务内，两次相同的范围查询，返回的行数不同（因为其他事务插入/删除并提交了）。

Q1：MySQL默认隔离级别是什么？解决了哪些问题？
MySQL InnoDB的默认隔离级别是 REPEATABLE READ（可重复读）。它通过MVCC解决了脏读和不可重复读问题。对于幻读，InnoDB在REPEATABLE READ下通过Next-Key Lock（记录锁+间隙锁）在加锁查询（如SELECT ... FOR UPDATE）时可以避免，但普通的快照读仍可能遇到幻读。所以MySQL的REPEATABLE READ实际上比SQL标准的要求更严格。


为了解决幻读，MySQL在REPEATABLE READ下使用了Next-Key Locking机制。
1）Next-Key Lock = Record Lock + Gap Lock：
-- 假设users表数据：
-- id: 1, 3, 5, 7, 9
-- age: 20, 25, 30, 35, 40

-- 事务1
START TRANSACTION;
SELECT * FROM users WHERE age = 30 FOR UPDATE;

-- Next-Key Lock锁定的范围：
-- 1. Record Lock：锁住age=30这行
-- 2. Gap Lock：锁住(25, 30)和(30, 35)这两个间隙
-- 3. 其他事务不能在(25, 35)范围内插入数据

-- 事务2尝试插入会被阻塞
START TRANSACTION;
INSERT INTO users (name, age) VALUES ('test', 28);  -- 阻塞，因为28在(25,30)间隙
INSERT INTO users (name, age) VALUES ('test', 32);  -- 阻塞，因为32在(30,35)间隙

2）不同查询条件的加锁策略
-- 示例数据：id (1,3,5,7), age (18,22,25,30)

-- 1. 等值查询（age = 25）
SELECT * FROM users WHERE age = 25 FOR UPDATE;
-- 加锁：record lock (age=25) + gap lock (22,25) + gap lock (25,30)

-- 2. 范围查询（age > 20）
SELECT * FROM users WHERE age > 20 FOR UPDATE;
-- 加锁：所有age>20的记录锁 + 所有相关间隙锁
-- 阻止在(20, ∞)范围内插入

-- 3. 无匹配记录的查询（age = 23）
SELECT * FROM users WHERE age = 23 FOR UPDATE;
-- 加锁：gap lock (22,25) 
-- 虽然没找到记录，但锁定了这个间隙

Q2：如何选择隔离级别？
读已提交（READ COMMITTED）：大多数Web应用的选择。平衡了性能和数据一致性，避免脏读，接受不可重复读和幻读。Oracle默认，推荐金融外的多数场景。

可重复读（REPEATABLE READ）：需要事务内数据一致性的场景，如对账、报表生成。MySQL默认。

串行化（SERIALIZABLE）：绝对数据一致性要求，如银行核心交易，但性能代价高。

Q3：什么场景下会用到 SERIALIZABLE？
两种典型场景：

余额检查与扣款：需要先SELECT查余额，再UPDATE扣款。在非串行化级别下，其他事务可能在两者之间修改余额，导致超额扣款。

唯一性复杂校验：需要查询多张表后插入，确保全局唯一。可用SERIALIZABLE配合SELECT ... FOR UPDATE，但实际更常用乐观锁或分布式锁替代，因为性能更好。

Q4：可重复读的隔离级别下：

事务A的操作	看到的数据版本	原因
SELECT（快照读）	1000（旧版本）	MVCC：使用事务开始时的读视图（根据MVVC，看到的是事务开始时的数据快照）
SELECT ... FOR UPDATE	900（新版本）	当前读：绕过MVCC，读取最新已提交数据并加锁
UPDATE	900（新版本）	当前读：必须基于最新数据修改（事务B修改后的900）

所以，在涉及余额扣减、库存减少等业务场景中，SELECT ... FOR UPDATE 不是可选项，而是必选项（建议 REPEATABLE READ + SELECT ... FOR UPDATE）

    public int factorial(int n) {
        if (n <= 1) return 1;  // ✅ 终止条件
        return n * factorial(n - 1);
    }
    
    // 2. 使用迭代替代递归
    public int factorialIterative(int n) {
        int result = 1;
        for (int i = 1; i <= n; i++) {
            result *= i;
        }
        return result;
    }

                used / 1024 / 1024,
                free / 1024 / 1024,
                max / 1024 / 1024,
                String.format("%.2f", usedPercentage));
    }
}

// JVM 内存结构（简化）

├─────────────────────────────────────┤

├─────────────────────────────────────┤

├─────────────────────────────────────┤

7.7 JIT：让 Java 飞起来的关键
JIT（Just-In-Time Compilation，即时编译） 是 JVM 的性能加速器


没有 JIT 时：
Java 字节码 → JVM 解释执行 → 有点慢

有 JIT 时：
热点代码（经常执行的代码） → JIT 编译成本地机器码 → 直接执行 → 很快！


JIT 就像给 Java 装了个记忆芯片，第一次执行时记住怎么做，以后直接照做，不用每次都重新想（解释）。这就是为什么 Java 越跑越快，而 PHP（传统方式）每次都要重新想。

@Override
protected void configure(HttpSecurity http) throws Exception {
    http.authorizeRequests()
        .antMatchers("/api/admin/**").hasRole("ADMIN") // /api/admin/ 下的接口需要ADMIN角色
        .antMatchers("/api/user/**").hasAnyRole("USER", "ADMIN") // 需要USER或ADMIN角色

9. ThreadLocal
9.1 为什么不能设计成“线程结束自动 remove”？
这个想法听起来很美好，但问题在于线程的生命周期并不总是和请求（业务逻辑）一致。

关键场景：线程池
在 Web 应用中，我们通常不会每次请求都创建新线程，而是使用线程池。线程池中的线程是复用的。一个线程处理完请求 A 后，会回到线程池，然后被分配给请求 B。

如果 ThreadLocal 在线程结束时自动清理，那它需要知道“线程什么时候结束”。但在线程池场景下，线程结束后不会销毁，而是被复用。所以，这个“结束”信号对线程池中的线程来说，是不存在的。

9.2 那为什么还需要手动 remove？
手动 remove 的目的，正是为了解决“线程复用导致的数据残留”问题。

1. 请求 A 在线程 T 中执行，往 ThreadLocal 里存了数据。
2. 请求 A 处理完毕，线程 T 回到线程池。
3. 如果不调用 remove，线程 T 的 ThreadLocalMap 里依然保留着请求 A 的数据。
4. 请求 B 被分配给线程 T 执行，它调用 ThreadLocal.get()，会错误地获取到请求 A 的数据，导致业务逻辑错误。

所以，手动 remove 的目的不是为了防止内存泄漏，而是为了防止“数据污染”——即确保下一次复用该线程时，不会读到上次残留的数据。

9.3 内存泄漏又是怎么回事？
ThreadLocal 确实有内存泄漏的风险，但关键在于键的引用方式。

ThreadLocalMap 的键是 ThreadLocal 对象本身，这是一个弱引用（WeakReference）。

如果 ThreadLocal 对象本身不再被任何强引用指向（比如你定义的 static ThreadLocal 超出了作用域），那么它会被 GC 回收。

此时，ThreadLocalMap 中的键变成了 null，但值（Value）仍然存在，且因为线程还在运行，这个值无法被回收，这就造成了内存泄漏。

所以，内存泄漏的真正原因是“键被回收了，但值还在”，而不是“线程结束没清理”。

9.4 Spring Boot Web API有没有一个跟随请求的ThreadLocal，这样就不会忘记remove？
1）RequestContextHolder：Spring 内置的请求级 ThreadLocal
RequestContextHolder 是 Spring 提供的一个工具类，它内部使用 ThreadLocal 存储了当前请求的 RequestAttributes（包含 HttpServletRequest、HttpServletResponse 等）。

// 在任何地方获取当前请求
ServletRequestAttributes attributes = 
    (ServletRequestAttributes) RequestContextHolder.getRequestAttributes();
HttpServletRequest request = attributes.getRequest();

关键在于：Spring 在 DispatcherServlet 处理请求时，会调用 RequestContextHolder.setRequestAttributes() 将请求绑定到当前线程，并在请求处理完成后，在 finally 块中自动调用 RequestContextHolder.resetRequestAttributes() 清理。

所以，你不需要手动 remove，Spring 已经帮你做了。

2）如何利用这个机制存储自定义数据？
自定义 ThreadLocal + 过滤器（推荐）
如果你需要存储与请求相关但非 HTTP 的数据（比如当前用户对象、权限列表、TraceId），更常见的做法是自定义一个 ThreadLocal，但在过滤器中统一管理生命周期。

@Component
public class UserContext {
    private static final ThreadLocal<User> currentUser = new ThreadLocal<>();

    public static void setUser(User user) {
        currentUser.set(user);
    }

    public static User getUser() {
        return currentUser.get();
    }

    public static void clear() {
        currentUser.remove();
    }
}

@Component
public class UserContextFilter implements Filter {
    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) {
        try {
            // 从 Token 或 Session 中获取用户信息
            User user = extractUserFromRequest(request);
            UserContext.setUser(user);
            chain.doFilter(request, response);
        } finally {
            // 关键：在 finally 中清理，确保不会残留
            UserContext.clear();
        }
    }
}
